# ImpactVault M1-3 Design

**Date:** 2026-03-10
**Status:** Approved (v2 вЂ” fork-based)
**Scope:** Months 1-3 deliverables

## Decisions

- **Location:** ~/Projects/impactvault/
- **Base:** Fork of OpenCheir (Rust MCP server)
- **Architecture:** Single binary with domain modules + nested Foundry project
- **Rust framework:** Axum (already in OpenCheir)
- **Smart contracts:** Foundry (Solidity)
- **Chain:** Sepolia testnet
- **Stablecoin protocol:** Aave V3 (Sepolia deployment)
- **RWA on testnet:** Mock ERC-4626 vault simulating sovereign bond yields

## Why Fork OpenCheir

OpenCheir provides proven infrastructure that maps directly to ImpactVault needs:

| OpenCheir Module | ImpactVault Use |
|-----------------|-----------------|
| Enforcer (rule engine) | Risk threshold enforcement, derisking rules |
| Lineage (audit trail) | Immutable log of health checks, derisking actions, disbursements |
| Patterns (discovery) | Systemic risk pattern detection |
| StateDb (SQLite) | Vault state, adapter configs, health history |
| Config (TOML + hot-reload) | Risk parameters, adapter configs, tunable without restart |
| MCP Server (rmcp) | Expose vault tools via MCP protocol |
| Gateway (router) | Prefix-based tool dispatch for vault/adapter/sentinel domains |

## Module Mapping

- `domain/sentinel.rs` вЂ” monitoring loop, health checks, auto-derisking

## Component Design

### 1. Core Risk Engine (`domain/engine.rs`)

Pure business logic. No network calls, no blockchain dependency.

**Key types:**

- `RiskSpectrum` вЂ” enum: `Sovereign`, `StablecoinSavings` (extensible)
- `VaultConfig` вЂ” approved yield sources, exposure caps, concentration limits, derisking thresholds
- `Portfolio` вЂ” current allocations across adapters, deposit/withdrawal/yield tracking
- `RiskAssessment` вЂ” health score, breach flags, recommended actions
- `AllocationPlan` вЂ” how to split a deposit across adapters
- `DeriskAction` вЂ” what to do when thresholds breach (Hold, Migrate, EmergencyWithdraw)

**Core functions:**

- `evaluate_risk(portfolio, health_data) -> RiskAssessment`
- `recommend_allocation(config, deposit_amount) -> AllocationPlan`
- `should_derisk(portfolio, health_checks) -> DeriskAction`

### 2. Adapter Layer (`domain/adapters/`)

Trait-based plugin system:

```rust
#[async_trait]
pub trait YieldAdapter: Send + Sync {
    fn name(&self) -> &str;
    fn risk_position(&self) -> RiskSpectrum;
    async fn deposit(&self, amount: U256) -> Result<TxRequest>;
    async fn withdraw(&self, amount: U256) -> Result<TxRequest>;
    async fn current_yield(&self) -> Result<f64>;
    async fn health_check(&self) -> Result<HealthStatus>;
    async fn tvl(&self) -> Result<U256>;
}
```

**M1-3 implementations:**

- `SovereignBondAdapter` вЂ” wraps MockRWAVault on Sepolia (~4-5% APY simulation)
- `AaveSavingsAdapter` вЂ” wraps Aave V3 on Sepolia (real testnet supply rates)

### 3. Smart Contracts (`contracts/`)

**ImpactVault.sol** вЂ” ERC-4626 vault with:
- Whitelisted depositors (gating)
- Configurable yield source routing
- Timelock on parameter changes
- Emergency derisk (can only reduce risk)

**YieldSplitter.sol** вЂ” yield distribution:
- Array of `(address, basisPoints)` recipient pairs
- On-chain event for every disbursement
- Admin updates via timelock

**MockRWAVault.sol** вЂ” testnet-only:
- ERC-4626 interface, mints synthetic yield at configurable APY

### 4. Sentinel (`domain/sentinel.rs`)

Async monitoring loop (reuses OpenCheir's enforcer + lineage):

- Polls adapter health checks on configurable interval (default 60s)
- Feeds results into enforcer rules (risk thresholds)
- On breach: engine's `should_derisk()` в†’ submit migration tx
- All checks logged via lineage (immutable audit trail)
- Patterns module discovers systemic risk patterns over time

### 5. MCP Tools (`gateway/server.rs`)

Exposed as `mcp__impactvault__<tool_name>`:

- `vault_status` вЂ” portfolio, allocations, health
- `vault_risk` вЂ” risk assessment
- `vault_deposit` вЂ” generate unsigned deposit tx
- `vault_withdraw` вЂ” generate unsigned withdraw tx
- `adapter_list` вЂ” available adapters and health
- `adapter_health` вЂ” detailed adapter health check
- `sentinel_status` вЂ” monitoring loop status
- `sentinel_history` вЂ” recent health check results
- `risk_evaluate` вЂ” run risk evaluation on demand
- `lineage_*` вЂ” inherited from OpenCheir
- `enforcer_*` вЂ” inherited from OpenCheir

### 6. REST API (`gateway/api.rs`)

Axum routes (testnet, no auth in M1-3):

- `GET /vault/{id}/status`
- `GET /vault/{id}/risk`
- `POST /vault/{id}/deposit`
- `POST /vault/{id}/withdraw`
- `GET /adapters`
- `GET /adapters/{name}/health`
- `GET /yield/history`
- `GET /disbursements`

### 7. Enforcer Rules (Risk-Specific)

Repurposed from OpenCheir's sliding-window rule engine:

- `derisk_on_health_breach` вЂ” trigger derisk if health score < threshold for N checks
- `oracle_staleness` вЂ” warn if oracle data older than threshold
- `concentration_limit` вЂ” block if single adapter exceeds % cap
- `liquidity_depth` вЂ” warn if available liquidity below threshold

Hot-reloadable via config.toml. Audit logged via lineage.
